Architecture

Hexagonal architecture (ports & adapters) with strict inward dependency flow. The core business logic never imports adapter code.

Hexagonal Architecture

nexusOrchestrator follows a hexagonal architecture (ports & adapters) with a strict inward dependency rule. Core services are isolated from infrastructure concerns — all external dependencies are behind interfaces (ports).

Inbound Adapters
HTTP API (chi router)  ·  MCP Server (JSON-RPC 2.0)  ·  CLI  ·  Wails Desktop GUI
↕
Core Services
OrchestratorService  ·  DiscoveryService  ·  ContextGuard
↕
Port Interfaces (Go interfaces)
OrchestratorPort  ·  LLMClientPort  ·  TaskRepositoryPort  ·  FileWriterPort  ·  SessionPort
↕
Outbound Adapters
LM Studio  ·  Ollama  ·  OpenAI  ·  Anthropic  ·  GitHub Copilot  ·  SQLite  ·  Filesystem
💡

Dependency rule: Inbound adapters → Core services → Ports ← Outbound adapters. The core never imports from adapters. All wiring happens in main.go via dependency injection.

Domain Layer

Pure Go types with no framework imports. These are the entities that flow through the entire system.

Task

The central entity representing a single unit of AI work:

Field Type Description
ID string UUID, generated on submission
ProjectPath string Absolute path to the project directory
TargetFile string Relative path for generated code output
Instruction string Natural language prompt sent to the LLM
ContextFiles []string Files included as context in the prompt
ModelID string Constrain to a specific model (optional)
ProviderHint string Prefer a specific provider by name (optional)
Command CommandType Task classification: plan, execute, or auto
Status TaskStatus Current lifecycle state
CreatedAt time.Time Creation timestamp
UpdatedAt time.Time Last update timestamp
Logs string LLM-generated output or error details

Provider

Represents an LLM backend with health state:

Field Type Description
Name string Human-readable provider name (e.g. "LM Studio")
BaseURL string API endpoint base URL
Active bool Whether the provider passed the last health check
ActiveModel string Currently loaded/preferred model ID
Models []string Available models (from /v1/models or equivalent)
APIKey string Authentication key (redacted in API responses)

TaskStatus Lifecycle

A task moves through a well-defined set of states from submission to completion:

QUEUED → PROCESSING → COMPLETED
PROCESSING → FAILED TOO_LARGE NO_PROVIDER QUEUED → CANCELLED
Status Description Terminal?
QUEUED Task is waiting in the queue for a free worker No
PROCESSING Task is being processed by an LLM backend No
COMPLETED LLM generated output successfully; result stored in Logs Yes
FAILED LLM call failed (network error, API error, etc.) Yes
CANCELLED Task was cancelled by the client before processing started Yes
TOO_LARGE Prompt exceeds the model's context window limit Yes
NO_PROVIDER No active provider available for the requested model Yes

Port Contracts

All ports are Go interfaces defined in the internal/ports/ package. Adapters implement these interfaces — the core never imports adapters directly.

OrchestratorPort

The primary inbound port — what the HTTP and MCP adapters call:

ports/orchestrator.go Go 
type OrchestratorPort interface {
    SubmitTask(ctx context.Context,
               req SubmitRequest) (domain.Task, error)
    GetTask(ctx context.Context,
            id string) (domain.Task, error)
    ListPendingTasks(ctx context.Context) ([]domain.Task, error)
    CancelTask(ctx context.Context,
               id string) error
    GetProviders(ctx context.Context) ([]domain.Provider, error)
    RegisterProvider(ctx context.Context,
                     p domain.Provider) error
    DeleteProvider(ctx context.Context,
                   name string) error
}

LLMClientPort

The outbound port for LLM backends — all providers implement this:

ports/llm.go Go 
type LLMClientPort interface {
    Complete(ctx context.Context,
             req CompletionRequest) (string, error)
    ListModels(ctx context.Context) ([]string, error)
    Health(ctx context.Context) error
    Name() string
}

TaskRepositoryPort

Persistence port for task storage (SQLite adapter):

ports/repository.go Go 
type TaskRepositoryPort interface {
    Save(ctx context.Context,
         task domain.Task) error
    FindByID(ctx context.Context,
             id string) (domain.Task, error)
    FindPending(ctx context.Context) ([]domain.Task, error)
    UpdateStatus(ctx context.Context,
                 id string,
                 status domain.TaskStatus,
                 logs string) error
}

SessionPort

Per-project conversation history port:

ports/session.go Go 
type SessionPort interface {
    GetHistory(ctx context.Context,
               projectPath string) ([]Message, error)
    AppendMessage(ctx context.Context,
                  projectPath string,
                  msg Message) error
    ClearHistory(ctx context.Context,
                 projectPath string) error
}

CommandType & Task Routing

Tasks can be classified to enable command-aware routing and enforcement:

plan

Plan Tasks

Orchestration, documentation, architecture decisions. These tasks produce plans that execute tasks must reference.

execute

Execute Tasks

Code implementation tasks. The orchestrator enforces that a prior plan task exists for the project before allowing execution.

auto

Auto (default)

The orchestrator infers the command type from the instruction content. Safe fallback for simple use cases.
